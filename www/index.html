Integrations in 3 steps

Nie zastępujemy WordPressa, WooCommerce, Baselinkera ani Allegro — spinamy je w jeden command center. Każda integracja to ten sam wzorzec: zdarzenie → URI → health → chat przy awarii.

1

Podłącz źródło

Webhook, REST API, plugin WordPress, cron lub skrypt — cokolwiek już masz. Taskinity woła to przez http:// lub shell://.

2

Zarejestruj proces

Jeden wpis w rejestrze deploymentów: nazwa, health URI, logi. Widzisz status w dashboardzie i API — bez grzebania po pięciu panelach.

3

Monitoruj i naprawiaj

Health co 5 min, alert webhookiem, chat po polsku: „co się stało z synchronizacją Allegro?”. Dry-run przed każdą mutacją.

Najprostszy wzorzec

Nie przepinasz firmy na nowy system. Dokładasz cienką warstwę nad tym, co już działa.

Każdy portal, ERP, CRM albo sklep dostaje jeden opis procesu: skąd przychodzi zdarzenie, jakie URI ma zostać wykonane, gdzie sprawdzić health i kto akceptuje kroki ryzykowne.

01 System źródłowy

WordPress, WooCommerce, BaseLinker, Allegro, ERP, CRM lub portal WWW.

02 Webhook / API

REST, cron, plugin, skrypt, Playwright albo adapter okienkowy.

03 URI workflow

workflow://order/sync, health://agent/..., repair://....

04 Chat i proof

Wynik, screenshot, logi, dry-run, approval i raport audytowy w jednym miejscu.

Minimalny kontrakt integracji

Najpierw nie budujesz pełnego konektora. Rejestrujesz jeden proces, jeden health check i jeden sposób diagnozy.

# 1. Opisz proces językiem biura
urish ask "połącz WooCommerce, BaseLinker i ERP; pokaż błędy w chacie"

# 2. Sprawdź plan bez skutków ubocznych
uri explain workflow://order/woocommerce-to-erp

# 3. Uruchom dopiero po approval
uri run workflow://order/woocommerce-to-erp --approve

# 4. Monitoruj i diagnozuj
uri call health://agent/invoices-agent.local
urish ask "dlaczego zamówienie z Allegro nie trafiło do ERP?"
Strony · API

Inne strony i systemy

  • Webhook POST /api/uri/call z Twojej aplikacji (formularz, SaaS, panel klienta).
  • Monitor URL co 5 min — alert, gdy strona lub endpoint padnie.
  • Reverse proxy (nginx) → health check na /health agenta.
curl -X POST http://localhost:8788/api/uri/call \
  -H "Content-Type: application/json" \
  -d '{"uri":"health://agent/invoices-agent.local","dry_run":true}'
ERP · CRM

Comarch · Subiekt · Salesforce · HubSpot

  • Cron lub n8n woła REST ERP → agent zapisuje wynik i logi błędów 401/500.
  • Przy awarii: incident z kontekstem (endpoint, payload, ostatni sukces).
  • Ticket do developera z gotowym planem naprawy — bez ręcznego kopiowania logów.
# przykład: faktura nie weszła do ERP
urish ask "zdiagnozuj agenta invoices-agent.local"
# → repair://agent/invoices-agent.local/diagnose
WordPress

WP · formularze · REST

  • Plugin lub functions.php: hook woocommerce_order_status_changed → webhook do Taskinity.
  • Application Password + REST API — odczyt statusu bez edycji core WP.
  • Formularz kontaktowy / lead → URI workflow (mail → CRM → potwierdzenie).
// WordPress → Taskinity (uproszczony hook)
wp_remote_post('http://taskinity.local/api/uri/call', [
  'body' => json_encode(['uri' => 'workflow://order/wp-to-crm', 'dry_run' => true]),
]);

Sklep · zamówienia · stany

  • Health na endpoint Woo REST — widać od razu, czy synchronizacja stoi.
  • Chat: „dlaczego zamówienie #4521 nie poszło do Baselinkera?”

Baselinker

Magazyn · marketplace hub

  • API Baselinker (getOrders, updateInventory) jako krok workflow URI.
  • Cron co N minut + porównanie liczby zamówień vs WooCommerce / Allegro.
  • Alert, gdy rozjazd stanów magazynowych przekroczy próg.
# shell:// lub python:// wrapper na API Baselinker
uri run workflow://sync/baselinker-inventory/dry-run
uri run workflow://sync/baselinker-inventory --approve
Allegro.pl

Aukcje · zamówienia · OAuth

  • Agent z tokenem OAuth Allegro — pobieranie zamówień, status wysyłki, zwroty.
  • Integracja z Baselinkerem lub bezpośrednio do ERP — jeden rejestr procesów.
  • Przy 429/503: supervisor retry z backoff, incident zamiast cichego failu.

How an office works — concrete examples

Marta, specjalista ds. rozliczeń, nie pisze skryptów. Opisuje zadanie po polsku w chacie. Taskinity planuje kroki: strona WWW, portal, okna w systemie, faktura, bank — a gdy trzeba tokenu na telefonie, prosi Martę o jeden klik i kontynuuje resztę.

Persona · pilot biurowy

Marta — rozliczenia i faktury

Codziennie: pobiera dane z portali, wystawia faktury w ERP, sprawdza PDF-y, robi przelewy w banku. Token mobilny akceptuje na Androidzie. Wcześniej traciła godziny na „dlaczego to znowu nie poszło”.

  • ~2 h/d powtarzalnych kliknięć
  • 5 systemów na raz
  • 1 chat zamiast pięciu okien

Co działa od razu, a co wymaga pilota

Gotowe w demo
  • Chat NL → plan URI → dry-run / approve
  • Health, logi, incident, ticket dla agentów
  • Przykłady: przeglądarka, Android (mock/ADB), Windows UIA (mock)
Wymaga konfiguracji u klienta
  • Logowanie do konkretnego banku / portalu (selektory, MFA)
  • Połączenie z Waszym ERP (Subiekt, Comarch, wFirma…)
  • Telefon Android podłączony przez ADB lub profil pracy

  • Token mobilny, SMS, biometria — Marta klika, system czeka
  • Wysyłka faktur / przelewów — approve przed mutacją
  • Audyt: kto zatwierdził, kiedy, jaki screenshot

Dla zespołów technicznych

URI to sekret architektoniczny — nie pierwsze hasło sprzedażowe.

Pod spodem każdy proces jest URI. To samo URI uruchomisz przez Web UI, API, CLI, workflow, shell, Docker lub SSH.

POST /api/ask          → pytanie NL → plan URI
POST /api/uri/call     → wykonanie (dry-run / approve)

$ urish proof view://process/invoice-sync/latest

Integracje MVP: http://, shell://, docker://. Deep tracing LLM → Langfuse / Phoenix / LangSmith (integracja, nie duplikacja).
